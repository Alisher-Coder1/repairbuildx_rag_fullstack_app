# STEP 1 — UI Contract Restoration v1

## Назначение файла

Этот файл фиксирует контракт пользовательского интерфейса для усиления production-версии проекта `repairbuildx_rag_fullstack_app`.

Текущая проблема проекта:

```text
UI собирает слишком мало данных
↓
backend получает бедный и частично неправильный контракт
↓
geometry engine считает слишком примитивно
↓
RAG не получает полноценный доменный контекст
↓
ответ консультанта выглядит как калькулятор
```

Цель Step 1:

```text
восстановить интерфейсную логику, близкую к Colab-версии,
но уже в production web-формате:
тип помещения → зона → форма → размеры → проёмы → покрытия → инженерные блоки → контекст ремонта → требования пользователя → консультация.
```

---

## 1. Главные сущности UI

Интерфейс должен собирать отдельные сущности:

```text
room_type       — тип помещения
zone_type       — зона эксплуатации
room_shape      — геометрическая форма помещения
dimensions      — размеры по выбранной форме
openings        — двери / окна / проёмы
surface_specs   — пол, стены, потолок
engineering     — инженерные блоки
repair_context  — состояние и уровень ремонта
user_goals      — требования пользователя
user_question   — вопрос консультанту
```

Эти сущности нельзя смешивать.

## 2. Тип помещения — `room_type`

Поле обязательное.

Допустимые значения v1:

```text
кухня
ванная
санузел
спальня
гостиная
коридор
прихожая
детская
кабинет
балкон
лоджия
кладовая
гардеробная
техническое помещение
```

`room_type` влияет на:

```text
рекомендуемую зону эксплуатации;
риски;
материалы;
инженерные требования;
структуру ответа консультанта.
```

Примеры:

```text
ванная → влажная зона → гидроизоляция, вентиляция, влагостойкие покрытия
кухня → кухонная зона → фартук, вытяжка, электрика, зона мойки
спальня → сухая зона → комфорт, акустика, покрытия без влажных требований
коридор → проходная сухая зона → износостойкость пола, грязевая нагрузка
балкон / лоджия → балконная зона → влажность, перепады температуры, утепление
```

---

## 3. Зона эксплуатации — `zone_type`

Поле обязательное, но должно подставляться автоматически по `room_type`.

### Автосвязка `room_type → zone_type`

```text
кухня → кухонная зона
ванная → влажная зона
санузел → влажная зона
спальня → сухая зона
гостиная → сухая зона
детская → сухая зона
кабинет → сухая зона
коридор → проходная сухая зона
прихожая → проходная сухая зона
балкон → балконная зона
лоджия → балконная зона
кладовая → сухая зона
гардеробная → сухая зона
техническое помещение → техническая зона
```

Пользователь может изменить зону вручную, но UI должен показать предупреждение:

```text
Вы изменили рекомендуемую зону. Проверьте, соответствует ли она условиям эксплуатации помещения.
```

## 4. Форма помещения — `room_shape`

Поле обязательное.

Допустимые значения:

```text
прямоугольная
круглая
Г-образная
сложная
```

Важно:

```text
room_shape управляет набором полей dimensions.
Нельзя показывать одинаковые поля размеров для всех форм.
Нельзя считать неподдерживаемую форму по формуле прямоугольника.
```

---

## 5. Размеры — `dimensions`

### 5.1. Прямоугольная комната

UI показывает:

```text
длина
ширина
высота
```

Payload:

```json
{
  "room_shape": "прямоугольная",
  "dimensions": {
    "length": 5.0,
    "width": 4.0,
    "height": 2.8
  }
}
```

Формулы:

```text
floor_area = length × width
ceiling_area = length × width
perimeter = 2 × (length + width)
wall_area_gross = perimeter × height
```

### 5.2. Круглая комната

UI показывает:

```text
диаметр
высота
```

UI не должен показывать `длина` и `ширина` как основные поля для круглой комнаты.

Payload:

```json
{
  "room_shape": "круглая",
  "dimensions": {
    "diameter": 6.0,
    "height": 2.8
  }
}
```

Формулы:

```text
radius = diameter / 2
floor_area = π × radius²
ceiling_area = π × radius²
perimeter = π × diameter
wall_area_gross = perimeter × height
```

### 5.3. Г-образная комната

Статус v1:

```text
future / disabled
```

Нельзя считать Г-образную комнату как прямоугольную.

Будущий payload:

```json
{
  "room_shape": "Г-образная",
  "dimensions": {
    "segments": [
      { "length": 4.0, "width": 3.0 },
      { "length": 2.0, "width": 2.0 }
    ],
    "height": 2.8
  }
}
```

Если пока не реализовано, UI должен показывать:

```text
Г-образная форма будет поддержана в следующей версии. Сейчас выберите прямоугольную или круглую форму.
```

### 5.4. Сложная форма

Статус v1:

```text
future / manual mode
```

Нельзя считать сложную форму как прямоугольную.

Будущий payload:

```json
{
  "room_shape": "сложная",
  "dimensions": {
    "manual_floor_area": 25.0,
    "manual_perimeter": 22.0,
    "height": 2.8
  }
}
```

## 6. Проёмы — `openings`

JSON-поле нельзя оставлять единственным способом ввода.

Основной UI должен быть таким:

```text
тип проёма: дверь / окно / арка / ниша
ширина
высота
количество
кнопка: добавить проём
список добавленных проёмов
кнопка: удалить
```

JSON можно оставить только как advanced mode.

Payload:

```json
"openings": [
  {
    "type": "дверь",
    "width": 0.9,
    "height": 2.1,
    "count": 1
  },
  {
    "type": "окно",
    "width": 1.5,
    "height": 1.4,
    "count": 1
  }
]
```

Проёмы влияют на:

```text
площадь стен;
чистую площадь стен;
длину плинтуса, если это дверь;
предупреждения по откосам и примыканиям.
```

## 8. Состояние помещения и уровень ремонта — `repair_context`

Блок обязательный.

Payload:

```json

  "property_condition": "черновая отделка",
  "repair_level": "капитальный",
  "has_existing_finish": false
}
```

Состояние помещения:

```text

```

```text

частичный
под ключ
только расчёт материалов

```

Это влияет на ответ консультанта.

## 13. Запрещённое поведение

```text
нельзя считать круглую комнату как прямоугольную;
нельзя показывать поля, которые backend не использует;
нельзя оставлять JSON-проёмы как единственный способ ввода;
нельзя смешивать room_type и room_shape;
нельзя делать вид, что неподдерживаемая форма поддерживается;
нельзя давать ответ только из чисел без инженерного вывода;
нельзя отправлять бедный payload, если UI визуально обещает больше;
нельзя использовать неподдерживаемую форму без явного warning/disabled state.
```

## 14. Критерии приёмки Step 1

Step 1 принят, если:

```text
UI собирает room_type;
UI автоматически предлагает zone_type;
room_shape управляет dimensions;
прямоугольная и круглая формы имеют разные поля;
Г-образная и сложная формы не считаются по ложной формуле;
проёмы вводятся через нормальный UI;
покрытия разделены по поверхностям;
есть состояние помещения;
есть уровень ремонта;
есть инженерные блоки;
есть пользовательские требования;
frontend payload соответствует UI Contract v1.
```
